# Contributing to Bemanning

Thank you for your interest in contributing to Bemanning! This document provides guidelines for contributing to this issues-only repository.

This is a **public issues-only repository** for the Bemanning web application. The main application code is maintained in a private repository. This repository is used exclusively for:

- Bug reports
- Feature requests
- Questions and support
- Community discussions

## üêõ How to Report Bugs

1. **Search existing issues** to see if your bug has already been reported
2. If not found, **create a new issue** using the Bug Report template
3. Provide as much detail as possible:
   - Clear description of the bug
   - Steps to reproduce
   - Expected vs. actual behavior
   - Screenshots if applicable
   - Browser and version information

## ‚ú® How to Request Features

1. **Search existing feature requests** to avoid duplicates
2. **Create a new issue** using the Feature Request template
3. Clearly describe:
   - The problem you're trying to solve
   - Your proposed solution
   - Who would benefit from this feature
   - Any alternatives you've considered

## üí¨ Asking Questions

For questions about using Bemanning:

1. **Check the documentation** first in our [Wiki](../../wiki)
2. **Search existing questions** in issues and discussions
3. For general questions, use [GitHub Discussions](../../discussions)
4. For specific support questions, create an issue using the Question template

- Share ideas and feedback

## üìã Issue Guidelines

### Good Issue Reports Include:

- **Clear title** that summarizes the issue
- **Detailed description** of the problem or request
- **Context** about your use case
- **Screenshots or examples** when applicable
- **Respectful tone** and constructive feedback

### Please Avoid:

- Duplicate issues (search first!)
- Vague descriptions without details
- Multiple unrelated topics in one issue
- Demanding or rude language

## üìû Getting Help

If you need help:

1. **Documentation**: Check our [Wiki](../../wiki)
2. **Discussions**: Ask in [GitHub Discussions](../../discussions)
3. **Issues**: Create an issue for specific bugs or questions

## ‚ö° Issue Triage Process

Issues go through the following process:

1. **Triage**: New issues are labeled and reviewed by maintainers
2. **Investigation**: Issues are analyzed and prioritized
3. **Planning**: Approved issues are added to development roadmap
4. **Implementation**: Features/fixes are implemented in the private repository
5. **Closure**: Issues are closed when resolved in a release

## üè∑Ô∏è Labels

We use labels to organize issues:

- `bug`: Something isn't working
- `enhancement`: New feature or request
- `question`: Question or support request
- `triage`: Needs review from maintainers
- `duplicate`: Duplicate of existing issue
- `wontfix`: Will not be addressed
- `in-progress`: Currently being worked on
- `help wanted`: Community help welcome

## üìù Response Times

While we strive to respond quickly:

- **Bug reports**: Usually reviewed within 2-3 business days
- **Feature requests**: Reviewed and prioritized regularly
- **Questions**: Answered as soon as possible

Please be patient, as this is maintained by a small team.
